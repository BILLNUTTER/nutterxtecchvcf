---
name: Deploying a pnpm-workspace Express+Vite app to Vercel
description: How to make a Replit artifact pair (separate Vite frontend + Express backend workflows) also deploy as one Vercel project.
---

When an app is built as two Replit artifacts (a Vite frontend service + an Express backend service in a pnpm workspace) but needs to also be Vercel-deployable as a single project:

- Vercel's zero-config "Other" builder only auto-detects Serverless Functions inside a **top-level `api/` directory relative to the project root** — not a nested path like `artifacts/<app>/api/`. If Root Directory stays at the repo root (common in a pnpm monorepo, since subfolder `vercel.json`/settings are only read if Root Directory is explicitly pointed there), the function file must live at the actual repo root (`api/index.ts`), and `vercel.json`'s `outputDirectory` must be the full path from the repo root to the built static assets (e.g. `artifacts/<app>/dist/public`).
- Do **not** have the Vercel function import the backend's raw TypeScript source (e.g. `@workspace/api-server/src/app`). Vercel's Node.js function builder enforces strict Node16/NodeNext ESM module resolution when type-checking/bundling `.ts` functions, which rejects extension-less relative imports (`./app` instead of `./app.js`) that are perfectly valid under a monorepo's `"moduleResolution": "bundler"` setting — this surfaces as confusing cascading errors like "Property 'use' does not exist on type 'Express'" or "not callable", because the wrong module-resolution mode misreads the type shape entirely.
- Fix: give the backend an esbuild-bundled "serverless" entry point (same bundler already used for its own production build) that exports just `{ app, connectMongo }` with no `app.listen()`, and have `api/index.ts` import that **compiled `.mjs` bundle**, not the TS source. Hand-write a sibling `.d.mts` next to the compiled output (via the build script) so TypeScript has types for the import without walking back into the original source tree.
- A `vercel.json` `functions` config entry needs at least one property (e.g. `maxDuration`) or Vercel errors with "Function must contain at least one property"; and the `runtime` field (if used at all) must be `name@version` format, not a bare string like `nodejs20.x`.
- Vite configs that hard-require `PORT`/`BASE_PATH` env vars (common in Replit scaffolds) must fall back to defaults when not serving (check `npm_lifecycle_event` for `dev`/`serve`) — this applies to *every* workspace package with a `build` script, not just the one being deployed, since the monorepo's root `pnpm run build` builds all of them.

**Why:** this lets one repository serve both Replit (multiple workflows/services) and Vercel (one project, one serverless function) without duplicating business logic, and avoids Vercel's TypeScript function builder silently applying different module-resolution rules than the rest of the monorepo.
